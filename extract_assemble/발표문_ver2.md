# 한국문학비평 TEI 자동 태깅 — ver2 발표문

## 0. 한 줄 요약
ver2는 **LLM이 XML을 만들지 않고 "의미 라벨"만 내고, 코드가 원문에 태그를 끼워 유효한 TEI를 조립**하는 구조입니다. 그 결과 ver1을 괴롭히던 **분량 손실과 스키마 위반이 구조적으로 사라졌고**, 정확도도 동등 이상으로 올랐습니다.

---

## 1. 문제의식 — 왜 ver2인가

ver1은 **LLM이 완성된 TEI/XML을 통째로 생성**하고 코드가 사후 수리하는 방식이었습니다. 두 가지 고질병이 있었습니다.

- **분량 손실**: LLM이 16,000자 본문을 매번 다시 출력해야 하니, 청크 경계에서 내용을 흘렸습니다. 실제로 한 실행에서는 **섹션 하나(전체의 28%)가 통째로 누락**되고, 출력이 잘려 **본문의 36%만 남은 적**도 있었습니다.
- **스키마 위반**: 같은 출력에서 **56건의 XSD 위반**이 나왔습니다 — 인라인 date(22건), 중복 xml:id(17건), interp·quote의 태그 중첩(17건) 등.

핵심 원인은 하나입니다. **"구조를 만드는 주체가 LLM"** 이라는 점. LLM은 비결정적이고 긴 출력에서 실수하므로, 구조와 분량을 보장할 수 없습니다.

---

## 2. 핵심 아이디어 — 추출-후-조립(extract-then-assemble)

발상을 뒤집었습니다.

| | ver1 (생성-수리) | **ver2 (추출-조립)** |
|---|---|---|
| LLM이 만드는 것 | 완전한 TEI/XML | **의미 라벨(JSON)만** |
| 구조를 만드는 주체 | LLM | **코드** |
| 비유 | "LLM이 글을 쓰고 코드가 교정" | **"LLM이 가리키고 코드가 받아 적음"** |

LLM의 역할을 **"이 부분이 인물, 이 부분이 연도"** 라고 지목하는 것으로 줄이고, **원문에 태그를 끼워 넣어 유효한 문서를 만드는 일은 코드**가 맡습니다. 코드가 원문을 절대 바꾸지 않으므로 **분량 손실이 불가능**하고, 코드가 스키마 규칙대로만 조립하므로 **위반도 불가능**합니다.

---

## 3. 처리 흐름

```
원문 텍스트
  │
  ├─ structure.py (코드)  원문 → 섹션·문단·문장 골격     ← 분량 100% 보존
  │
  ├─ annotate.py  (LLM)   각 문장 → 개체 라벨 JSON
  │                        [{text:"최백규", tag:"persName", attrs:{role:"poet"}}, …]
  │
  ├─ resolve.py   (LLM 1회) 전역 인물 role 통일
  │
  └─ assemble.py  (코드)   골격 + 라벨 → 유효한 TEI + XSD 검증
```

- **structure**: PDF에서 추출한 원문을 섹션·문단·문장으로 나눕니다. (PDF의 줄바꿈이 단어 중간을 끊는 문제도 여기서 바로잡았습니다.)
- **annotate**: 문장마다 LLM이 개체를 라벨링합니다. 문장 텍스트는 절대 바꾸지 않고 그 안의 부분 문자열만 지목합니다.
- **assemble**: 라벨을 원문 위에 얹어 TEI를 조립하고, XSD로 즉시 검증합니다.

---

## 4. 스키마를 "코드로 강제"한다

조립 단계에서 ver1의 56건 위반을 **유형별로 원천 차단**했습니다.

| ver1의 위반 | ver2의 강제 방식 |
|---|---|
| 인라인 date 위치 | 정책대로 처리 (v5 스키마에선 정식 허용) |
| 중복 xml:id | 첫 등장만 `xml:id`, 반복은 `ref="#…"` → 항상 유일 |
| interp/quote 태그 중첩 | 겹치는 span은 코드가 비중첩만 선택 |
| 속성 enum 위반 | role·level·type 등을 허용값으로 스냅, 아니면 제거 |
| 헤더 누락(respStmt 등) | 코드가 완전한 teiHeader 고정 조립 |

**스키마(XSD)가 곧 설계도이고, 코드가 그 설계도를 강제하는 검증기 역할**을 합니다.

### v5 스키마 반영 — date
발표 준비 중 스키마를 v5로 확장하면서 **date가 본문 문장에 정식 허용**되었고, 연도/연대를 정밀하게 표기하게 했습니다.
- 단일 연도: `<date when="2022">`
- 10년대(기간): `<date from="2010" to="2019">`

이는 정답지(수작업 태깅)의 관행과 정확히 일치합니다.

---

## 5. 성과 (정답지 대비)

수작업으로 태깅한 **정답지(gold)** 를 기준으로 자동 태깅을 채점했습니다.

- ✅ **XSD 검증: 0 위반** (v5 스키마 완전 통과)
- ✅ **분량: 문장 100% / 글자 100%** (원문 한 글자도 안 빠짐)
- ✅ **정확도(F1): micro 느슨 68.8% / 엄격 65.0%**
  - 작품 87.9% · 기관 90.0% · 인물 85.9%(찾기) 등은 높은 수준

여기서 **느슨 = "개체를 찾았는가", 엄격 = "속성(role/type 등)까지 맞췄는가"** 입니다.

---

## 6. 핵심 개선 사례 — 인물 역할(role)

가장 큰 개선이 있었던 부분입니다.

- **문제**: 인물은 잘 찾는데(85.9%) 역할은 거의 다 틀렸습니다(엄격 **12.5%**).
- **원인 진단**: 역할을 잘못 단 게 아니라, **정답지는 role을 '첫 등장에만' 달고 반복은 `ref`로 참조**하는데, 자동 태깅은 모든 언급에 role을 달아 어긋난 것이었습니다. 또 외국 이론가(마크 피셔·낸시 프레이저 등)를 `scholar`로 단 점도 있었습니다.
- **해결**: ① LLM에게 인물 목록을 한 번에 주고 **전역적으로 역할 통일**(외국 이론가→`foreigner`), ② 조립기에서 **첫 등장만 role, 반복은 `ref`만** 달도록 관행 일치.
- **결과**: 인물 역할 엄격 **12.5% → 79.7%**, 전체 엄격 F1 **54% → 65%**.

> 시사점: ver2에서는 **데이터로 원인을 진단하고, 코드 규칙으로 정밀하게 교정**하는 개선이 가능합니다.

---

## 7. ver2의 또 다른 장점 — "안전한 프롬프트 엔지니어링"

ver1에서는 "용어를 더 많이 태깅하라"고 프롬프트를 강화하면 **출력이 길어져 잘리고 분량이 무너졌습니다**(실제로 겪음).

ver2에서는 LLM이 **라벨만** 내므로, 프롬프트를 아무리 공격적으로 바꿔도 **분량·유효성은 코드가 보장**합니다. 즉 **프롬프트 실험의 위험이 사라져**, 정답지를 기준으로 마음껏 튜닝하고 즉시 채점하는 루프가 가능해졌습니다.

---

## 8. 한계와 향후

정직하게, 모든 게 완벽하진 않습니다.

- **용어(term)·해석(interp)**: 용어는 사람마다 태깅 기준이 다르고, 해석은 본질적으로 주관적입니다(어디까지가 '판단'인가). → **최종 사람 검수** 영역으로 둡니다.
- **인용(quote)**: 정답지는 시 인용을 운문 구조(`<l>`·`<lg>`)로 마크업하는데 자동은 평문으로 둡니다. 구조 정합은 향후 과제입니다.
- 시도했다 폐기한 것: **사전 힌트로 용어 보강** — 사전에 일반어가 많아 과태깅을 유발해 되돌렸습니다(정직한 음성 결과).

---

## 9. 마무리

ver2는 **"LLM에게 무엇을 맡기고, 무엇을 코드에 맡길 것인가"** 를 다시 나눈 결과입니다.

- LLM에게는 **의미 판단**(무엇이 인물·작품·용어인가)을,
- 코드에게는 **구조·유효성·분량 보장**을 맡겼습니다.

그 결과 **유효하고 완전한 TEI를 안정적으로** 생산하면서, 정확도도 끌어올리고, 검증 가능한 개선 루프를 갖추게 되었습니다. 남은 부분은 사람 검수와 명확히 분리되어, **자동화와 전문가 검수가 협업하는 구조**가 완성되었습니다.

감사합니다.
