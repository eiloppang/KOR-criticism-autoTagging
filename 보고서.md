# KorCritTEI 개발 보고서

**프로젝트**: 한국문학비평 TEI 자동 태깅 앱 (KorCritTEI)
**작성일**: 2026-03-25
**대상 파일**: `core/tagger.py`, `core/dictionary_matcher.py`, `app.py`, `ui/tagging_tab.py`

---

## 1. 프로젝트 개요

KorCritTEI는 한국 근현대 문학비평 텍스트를 TEI/XML로 자동 변환하는 Streamlit 앱이다. LLM(Gemini / Claude / Ollama)을 활용하여 비평문에서 인물명·작품명·기관명·용어·인용문·해석·날짜를 태깅하고, `korean-critique-schema.xsd` 스키마에 대해 XSD 검증을 수행한다.

### 핵심 처리 파이프라인

```
입력 텍스트
  │
  ├─ AdaptiveChunker  →  청크 분할 (기본 5,000자, 단락 경계 존중)
  │
  ├─ DictionaryMatcher →  사전 매칭 → [사전 매칭 참고자료] 블록 생성
  │
  ├─ LLM (Gemini/Claude/Ollama)
  │     ├─ 청크 1: 완전한 TEI 문서 (teiHeader 포함)
  │     └─ 청크 2+: text > body > div 구조만 출력
  │
  ├─ _merge_body_divs  →  청크 병합 (div 순서 결합)
  │
  ├─ _finalize_tei     →  후처리 (teiHeader 재구성 + xml:id 중복 통일)
  │
  └─ validate_xml      →  XSD 스키마 검증
```

## 2. 이번 세션에서 변경된 사항

### 2-1. 네임스페이스 접두사 문제 수정 — `core/tagger.py`

**문제**: `_merge_body_divs()`에서 병합 후 직렬화할 때 `<ns0:TEI>`, `<ns0:persName>` 형태의 접두사가 출력됨.

**원인**: lxml에서 `etree.Element(f"{{{NS}}}TEI")`로 루트 요소를 생성할 때 기본 네임스페이스(`xmlns=`) 선언이 없으면 자동으로 `ns0:` 접두사를 부여함.

**수정**:
```python
# 수정 전
tei = etree.Element(f"{{{NS}}}TEI")

# 수정 후
tei = etree.Element(f"{{{NS}}}TEI", nsmap={None: NS})
```

`nsmap={None: NS}`로 기본 네임스페이스를 명시하면 접두사 없이 `<TEI xmlns="http://www.tei-c.org/ns/1.0">` 형태로 직렬화된다.

### 2-2. 사전 매칭 모듈 신규 구현 — `core/dictionary_matcher.py`

세 종의 CSV 사전 파일을 로드하고 비평텍스트에서 인물·용어 엔티티를 매칭하여 LLM 프롬프트에 참고자료를 삽입하는 모듈을 새로 작성했다.

#### 주요 클래스 및 함수

| 구성 요소 | 역할 |
|---|---|
| `DictionaryMatcher` | 사전 로드·인덱스 관리·매칭·포맷 통합 클래스 |
| `load_from_dir(dict_dir)` | `dict/` 폴더의 3개 CSV 자동 로드 |
| `load_from_uploaded(files)` | Streamlit UploadedFile 리스트 수신 |
| `match(text, fuzzy_threshold)` | 완전 일치(substring) + RapidFuzz 퍼지 매칭 |
| `format_for_prompt(matches)` | `[사전 매칭 참고자료]` 블록 생성 |
| `load_default_matcher(dict_dir)` | 편의 함수 — 기본 경로로 matcher 반환 |

#### 매칭 전략

1. **완전 일치**: 각 headword / alias가 텍스트 내에 부분 문자열로 존재하는지 검사.
2. **퍼지 매칭**: 텍스트를 공백·구두점 기준으로 토큰화 후 RapidFuzz `fuzz.ratio`로 유사도 비교. `rapidfuzz` 미설치 시 자동으로 완전 일치만 수행.

#### 엔티티 유형 구분

`gender`, `birth_year`, `death_year` 컬럼 중 하나라도 값이 있으면 **인물**로 판정, 없으면 **용어·개념**으로 분류.

#### 프롬프트 삽입 형식 예시

```
[사전 매칭 참고자료]
아래는 비평텍스트에서 발견된 인물 및 용어의 사전 정보입니다...

## 인물
- **박상수** | 1970~? | 시대: 현대 | 분야: 소설가 | ref: https://...

## 용어·개념
- **서사** | 일반적으로 서사(敍事)는 어떤 사실을 있는 그대로 기록하는… | ref: https://...
```

### 2-5. xml:id 중복 통일 — `_dedup_xml_ids()` in `core/tagger.py`

**문제**: 청크를 병합할 때 동일한 인물/작품이 청크마다 다른 `xml:id`를 부여받음.
예: 청크 1에서 `p-parksangsu`, 청크 2에서 `p-parksangsu2`, 청크 3에서 `p-parksangsu3`.

**알고리즘**:

```
Pass 1: (태그명, 정규화텍스트) → 등장한 xml:id 목록 수집 (문서 순)
        예: ("persName", "박상수") → ["p-parksangsu", "p-parksangsu2", "p-parksangsu3"]

canonical 선택:
  ① 숫자 미접미사 id 우선  →  p-parksangsu  ✓
  ② 숫자 미접미사 없으면 첫 번째 id 사용

Pass 2: 모든 대상 요소에 canonical id 일괄 적용
```

대상 요소: `persName`, `title`, `orgName`

### 2-6. taxonomy 고정 삽입 및 teiHeader 재구성 — `core/tagger.py`

**문제**:
1. 청크마다 LLM이 `classDecl`을 다르게 생성하여 taxonomy가 일관되지 않음.
2. 병합 과정에서 `teiHeader`가 완전히 사라지는 케이스 발생.

**근본 원인 분석**: `_fix_taxonomy()`는 teiHeader가 이미 존재할 때만 encodingDesc를 추가하므로, teiHeader가 None인 경우 아무것도 삽입하지 않고 반환했다. lxml의 요소 이동(다른 트리에서 append) 중 일부 경우에 header_el이 소실되는 문제도 존재했다.

**해결 전략**: teiHeader를 LLM 출력에 의존하지 않고 **코드에서 완전히 재구성**한다.

#### 신규 함수 구성

| 함수 | 역할 |
|---|---|
| `_extract_header_meta(header_el, NS)` | 기존 teiHeader에서 title / author 텍스트 추출. None이면 빈 문자열 반환 |
| `_build_tei_header(title, author)` | lxml로 완전한 teiHeader 직접 구성 |
| `_finalize_tei(xml_str)` | 통합 후처리 진입점. 단일·다중 청크 모두 적용 |

#### `_build_tei_header()` 출력 구조

```xml
<teiHeader>
  <fileDesc>
    <titleStmt>
      <title><!-- LLM 첫 청크 추출값 또는 "미상" --></title>
      <author><!-- LLM 첫 청크 추출값 또는 "미상" --></author>
    </titleStmt>
    <publicationStmt><p>디지털 인문학 연구 목적</p></publicationStmt>
    <sourceDesc><p>입력 텍스트</p></sourceDesc>
  </fileDesc>
  <encodingDesc>
    <classDecl>
      <taxonomy xml:id="tax-role">
        <category xml:id="critic"/>
        <category xml:id="poet"/>
        <category xml:id="novelist"/>
        <!-- ... 총 10개 역할 -->
      </taxonomy>
      <taxonomy xml:id="tax-interp">
        <category xml:id="affirmative"/>
        <category xml:id="neutral"/>
        <category xml:id="critical"/>
      </taxonomy>
    </classDecl>
  </encodingDesc>
</teiHeader>
```

#### `_finalize_tei()` 처리 순서

```
1. XML 파싱 (실패 시 원본 반환)
2. 기존 teiHeader에서 title/author 추출
3. 기존 teiHeader 제거
4. _build_tei_header()로 새 teiHeader 생성 → root.insert(0, ...) 삽입
5. _dedup_xml_ids() 실행
6. 기본 네임스페이스 검사 (ns0: 접두사 방지)
7. cleanup_namespaces() + 직렬화
```

이 설계로 **teiHeader 누락이 구조적으로 불가능**해졌다. LLM이 teiHeader를 생성하지 않거나 파싱에 실패해도 코드 측에서 반드시 삽입한다.

## 5. 이후 개선 고려사항

### 5-1. XML 품질 개선

**xml:id 교차 참조 업데이트**
`_dedup_xml_ids()`는 현재 `persName`/`title`/`orgName` 요소의 `xml:id` 속성만 통일한다. 문서 내에 `#p-parksangsu2` 형태의 참조가 `ana`, `ref`, `corresp` 등 다른 속성에 존재하면 깨진 참조가 남는다. 전체 속성 값을 스캔하는 참조 업데이트 로직이 필요하다.

**teiHeader 메타데이터 보강**
현재 `title`과 `author`만 추출한다. 비평문 발표 연도·출처 잡지명(`sourceDesc`)을 사전 매칭 결과나 텍스트에서 자동 추출하면 서지 정보가 더 풍부해진다.

**respStmt 자동 생성**
TEI 표준에서 `respStmt`는 인코딩 책임자 정보를 기술한다. 현재는 생략되어 있으며, 앱 사용자 정보나 LLM 모델 정보를 자동 삽입하면 출처 추적이 용이해진다.

---

### 5-2. 사전 매칭 개선

**형태소 분석 기반 매칭**
현재는 공백·구두점 기반 경량 토크나이저를 사용한다. `kiwipiepy` 등 한국어 형태소 분석기를 연동하면 활용형 매칭 정확도가 높아진다 (예: "박상수의" → "박상수" 매칭).

**매칭 결과 UI 표시**
태깅 탭에서 어떤 사전 엔트리가 몇 개 매칭됐는지 테이블로 보여주면 사용자가 참고자료 품질을 직접 확인할 수 있다.

**사전 정규 업데이트 지원**
현재 사전 CSV는 `dict/` 폴더에 정적으로 존재한다. 버전 관리나 갱신 날짜 추적 메커니즘을 추가하면 장기적인 유지보수가 쉬워진다.

**엔티티 중의성 해소**
동명이인(같은 이름, 다른 인물)은 현재 첫 번째 매칭 결과를 사용한다. 시대·분야 컨텍스트를 활용한 중의성 해소 로직이 필요하다.

---

### 5-3. 청킹 및 LLM 파이프라인 개선

**청크 간 태그 연속성 보장**
현재 각 청크는 독립적으로 태깅되므로 같은 인물이 청크 경계를 넘어 처음 등장할 때 다른 `xml:id`를 받는다. `_dedup_xml_ids()`로 사후 통일하지만, 근본적으로는 이전 청크의 엔티티 목록을 다음 청크 프롬프트에 주입하는 **엔티티 컨텍스트 전파** 방식이 더 정확하다.

**병렬 청크 처리**
현재는 청크를 순차 처리한다. 2번째 이후 청크는 상호 의존성이 낮으므로 `asyncio` 또는 `ThreadPoolExecutor`로 병렬화하면 처리 속도를 크게 줄일 수 있다.

**스트리밍 출력**
LLM API의 스트리밍 기능을 활용하면 긴 청크 처리 중 사용자에게 실시간 진행 상황을 보여줄 수 있다.

---

### 5-4. 검증 및 품질 관리

**XSD 검증 오류 자동 수정**
현재는 XSD 오류를 경고로 표시하는 데 그친다. 자주 발생하는 오류 유형(잘못된 `level` 값, 필수 속성 누락 등)을 탐지하여 자동 교정하는 포스트프로세서를 추가할 수 있다.

**태깅 재현성 평가 도구**
같은 텍스트를 반복 태깅했을 때 결과의 일관성을 측정하는 간단한 평가 모듈(F1 스코어 등)을 추가하면 LLM 모델·파라미터 선택에 객관적 근거가 생긴다.

---

### 5-5. UI / 배포

**결과 비교 뷰**
원문 텍스트와 태깅 결과 XML을 나란히 보여주는 split view를 `ui/tagging_tab.py`에 추가하면 검수 편의성이 크게 향상된다.

**배치 처리 모드**
현재는 단일 파일만 처리한다. 여러 비평문을 한 번에 업로드하여 순차 처리하고 ZIP으로 다운로드하는 배치 모드가 있으면 대량 처리에 유용하다.

**Docker 컨테이너화**
`requirements.txt`와 `Dockerfile`을 정비하여 배포 환경을 표준화하면 협업자가 동일한 환경을 쉽게 재현할 수 있다.
